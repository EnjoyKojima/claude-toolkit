# GitHub Issue 実装スキル

GitHub Issueを起点に、仕様の深掘り → Issue整理 → 実装計画 → 実装 の一連のフローを進める。

## フロー全体像

```
1. Issue読み込み
2. 仕様グリル（常に発火）
3. Issue整理（分割・スコープ外Issue提案）
4. 実装計画の提示（ユーザー視点の動作説明）
5. ユーザー承認
6. 実装
```

---

## ステップ1: Issue読み込み

### 引数の解釈

- `$ARGUMENTS` がある場合: Issue番号またはURLとして `gh issue view` で内容を取得
- `$ARGUMENTS` がない場合: `AskUserQuestion` で「どのIssueに取り組みますか？」と聞く

### 取得する情報

```bash
gh issue view <number> --json title,body,labels,assignees,milestone,comments
```

Issueの内容を読み込んだら、タイトルと概要を表示してからステップ2に進む。

---

## ステップ2: 仕様グリル

**常に発火する。スキップしない。**

Issueの内容をもとに、実装に必要な仕様の曖昧な部分・未決定事項を洗い出し、1問ずつ徹底的にインタビューする。

### 質問のスタイル

1. **質問の背景と各選択肢の根拠を先に説明する**
   - なぜこの質問が必要なのか
   - 各選択肢を選んだ場合にどうなるか（メリット・デメリット・影響範囲）
   - 推奨する選択肢とその理由

2. **説明した後に `AskUserQuestion` を発火する**
   - 選択肢を事前に定義し、ユーザーがキーボードの上下キーで選べるようにする
   - 自由記述が必要な場合は選択肢の最後に「その他（自由入力）」を含める

3. **1問ずつ聞く**
   - 前の回答を踏まえて次の質問を深掘りする
   - 回答から新たな疑問が生まれたら追加質問する

### 質問の観点

- Issueに書かれていない暗黙の前提条件
- エッジケースの扱い（空データ、上限値、権限境界）
- 既存機能への影響範囲
- ユーザーから見た振る舞いの期待値
- エラー時のふるまい
- 実装方針で複数のアプローチがあり得る場合の選択

### コードベース探索

質問に答えるために既存コードを読めば分かることは、**自分で調べて自ら答えを出す。ユーザーに聞かない。**
ユーザーに聞くのは、コードからは判断できないビジネス上の意思決定のみ。

### グリルの終了条件

- 全ての曖昧な点が解消された
- ユーザーが「もう十分」「実装して」と明示的に終了を指示した

---

## ステップ3: Issue整理

グリルで明らかになった情報をもとに、Issueの整理を提案する。

### 3a. サブIssue分割

元のIssueが複数の関心事を含む場合、以下の3レイヤーでの分割を基本として提案する:

| レイヤー | スコープ |
|---|---|
| **API** | GraphQLスキーマ・DBスキーマ・サービス層・リゾルバー（admin + storefront両方） |
| **ストアフロントUI** | ストアフロント側のフロントエンド実装 |
| **管理画面UI** | 管理画面側のフロントエンド実装 |

**注意:**
- 全てのIssueが3分割になるわけではない。APIだけ、UIだけのIssueもある
- API側の変更が大きい場合、admin/storefrontでさらに分割する提案をしてもよい
- **分割は必ず提案ベース。** ユーザーの承認なしに勝手にIssueを作成しない
- 分割が不要（1つのPRで十分な規模）と判断した場合は、分割しない理由を説明する

### 3b. アウトオブスコープIssue

グリル中に発見したスコープ外の問題・改善点を整理する。

- **重要なもの**: グリル中にリアルタイムで「これはスコープ外ですが、Issue作りますか？」と都度確認済み
- **軽微なもの**: このタイミングでまとめてリスト提示し、作成するか確認する

いずれも `AskUserQuestion` でユーザーの承認を得てからIssue作成する。

---

## ステップ4: 実装計画の提示

**ユーザー視点で「何がどう変わるか」を簡潔に文章で説明する。**

### 書き方

- 技術的な詳細（ファイルパス、関数名、型定義など）は書かない
- 「今こうなっている」→「こう変わる」の形式で、動作の変化を説明する
- ユーザーが「それでいい」「違う」を判断できる粒度にする

### 例

```
現在、商品一覧ではタグによる絞り込みができません。
実装後は、ストアフロントの商品一覧画面でタグを選択すると、
そのタグが付いた商品だけが表示されるようになります。
管理画面側では、商品編集時にタグの追加・削除ができるようになります。
```

### 承認

計画を提示したら `AskUserQuestion` で承認を求める:
- 「この計画で実装を開始する」
- 「計画を修正したい」
- 「やめる」

---

## ステップ5: 実装

ユーザーの承認を得たら、プロジェクトのCLAUDE.mdに記載された開発ワークフローに従って実装を進める。

CLAUDE.mdが存在しない場合は一般的なベストプラクティスに従う。

---

## 注意事項

- このスキルの全てのユーザーへの確認は `AskUserQuestion`（選択肢付き）で行う
- 選択肢を提示する前に、必ず各選択肢の根拠・推奨理由を文章で説明する
- Issueを作成・分割する前に必ずユーザーの承認を得る
- 提案なしに勝手に判断しない。判断が必要な場面では常にユーザーに確認する
