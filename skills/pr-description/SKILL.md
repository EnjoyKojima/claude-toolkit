# PR説明文 生成・更新スキル

PRの実装内容を分析し、レビュワーとフロントエンド実装者の両方に役立つ説明文を生成する。

## 目的

- レビュワーが「何が変わったか」「なぜ変えたか」を素早く理解できる
- フロントエンド開発者がAPIの使い方をPR説明だけで把握できる
- 技術詳細が一覧で俯瞰できる

## ワークフロー

### 1. PR情報の取得

#### 引数の解釈

- `$ARGUMENTS` がある場合: PR番号またはURLとして `gh pr view` で内容を取得
- `$ARGUMENTS` がない場合: 現在のブランチに紐づくPRを `gh pr view` で取得。PRがなければ `AskUserQuestion` で聞く

#### 取得する情報

```bash
# PR基本情報
gh pr view <number> --json title,body,baseRefName,headRefName,url,number

# PR差分（全ファイル）
gh pr diff <number>

# PRコミット履歴
gh pr view <number> --json commits

# リンクされたIssue（あれば）
gh pr view <number> --json body
# bodyから "Closes #XXX" や "Fixes #XXX" を抽出
```

### 2. 実装内容の分析

PR差分とコミット履歴を読み、以下を整理する。

#### 2a. 変更の全体像

- **目的**: 何のための変更か（リンク先Issueがあればその内容も参照）
- **変更カテゴリ**: 新機能 / 機能改善 / バグ修正 / リファクタリング / インフラ / その他

#### 2b. レイヤー別の変更内容

差分から以下のレイヤーを特定する:

| レイヤー | 確認対象 |
|---|---|
| DDD | `docs/ddd/` 配下の変更 |
| DB | `packages/db/src/schema.ts`、マイグレーションファイル |
| GraphQL | `.graphql` スキーマファイルの変更 |
| Admin API | `backend/admin/` 配下の変更（リゾルバー・サービス） |
| Storefront API | `backend/storefront/` 配下の変更（リゾルバー・サービス） |
| インフラ | `infra/` 配下の変更 |
| シード | シードデータの変更 |
| フロント（Admin） | Admin フロントエンドの変更 |
| フロント（Storefront） | Storefront フロントエンドの変更 |

変更のないレイヤーは省略する。

#### 2c. GraphQL API変更の詳細分析（重要）

`.graphql` スキーマに変更がある場合、**フロントエンド開発者向けのAPIガイド** を作成するために以下を詳細に分析する:

- 追加・変更された Query / Mutation / Subscription
- 入力型（Input type）のフィールドとバリデーション
- レスポンス型のフィールド構成
- 認可ルール（`@aws_auth` ディレクティブ）
- サービス層のバリデーションロジック（エラーコード、上限値、文字数制限など）

**サービス層のテストコードも読む。** テストにはバリデーションルールやエッジケースの仕様が具体的に書かれていることが多い。

### 3. 説明文の生成

以下の構造で説明文を生成する。**全セクションが必須ではない。変更内容に応じて必要なセクションだけを含める。**

````markdown
## Summary

Closes #XXX（リンクIssueがあれば）

（1-3文で変更の概要を説明。「何を」「なぜ」を簡潔に）

---

### フロントエンド実装向けガイド

（GraphQL API変更がある場合のみこのセクションを含める）

#### Admin（管理画面）

（Admin側のAPI変更がある場合のみ）

**[操作名]**

```graphql
（実際に使えるクエリ/ミューテーション例）
```

- （動作の説明、バリデーションルール、権限、注意点をリストで）

#### Storefront（ストアフロント）

（Storefront側のAPI変更がある場合のみ）

**[操作名]**

```graphql
（実際に使えるクエリ/ミューテーション例）
```

- （動作の説明、注意点をリストで）

---

### 技術詳細

| レイヤー | 変更内容 |
|---------|---------|
| （変更のあったレイヤーのみ列挙） |

## Test plan

- [x] （実際に書かれた/通っているテストの概要）
- [x] `pnpm tc` 型チェック通過
- [x] `pnpm lint` エラーなし

🤖 Generated with [Claude Code](https://claude.com/claude-code)
````

### 生成ルール

#### フロントエンド実装向けガイド

- **GraphQLクエリ/ミューテーション例は実際に動くものを書く。** スキーマと実装を照合して正確な例を作る
- フィールド名はスキーマの定義通りに。推測で書かない
- バリデーションルール（文字数制限、上限数など）はサービス層のコードから正確に抽出する
- 権限（誰がどの操作を行えるか）を明記する
- Admin と Storefront で返却フィールドが異なる場合、その差分を明示する
- 一括操作の場合、その動作モデル（全件置換、差分更新など）を説明する

#### 技術詳細テーブル

- 各レイヤーの変更を1行で簡潔に説明する
- 変更のないレイヤーは含めない
- テーブル名、カラム名など具体的な名前を含める

#### テストプラン

- 実際にPRに含まれるテストの内容を反映する
- テストがない変更（インフラ、設定変更など）の場合は手動確認項目を書く
- `pnpm tc` と `pnpm lint` は常に含める

### 4. ユーザーへの提示と更新

生成した説明文をユーザーに表示し、`AskUserQuestion` で確認する:
- 「この内容でPRの説明を更新する」
- 「修正したい箇所がある」
- 「やめる」

「修正したい箇所がある」が選ばれた場合、具体的な修正指示を聞いて反映する。

承認されたら `gh pr edit <number> --body` で説明文を更新する:

```bash
gh pr edit <number> --body "$(cat <<'EOF'
（生成した説明文）
EOF
)"
```

更新完了後、PRのURLを表示する。

## 注意事項

- GraphQL例は必ず `.graphql` スキーマファイルと実装を照合して正確に書く。推測しない
- バリデーションルールはサービス層のコード（定数、条件分岐）から抽出する。テストコードも参考にする
- PRにテストが含まれていなくても、テストプランのセクションは含める（手動確認項目として）
- 説明文は日本語で書く
- 既存の説明文がある場合でも、実装内容を分析して新たに生成する（既存の説明文が実装と乖離している可能性があるため）
- 説明文の更新前に必ずユーザーの確認を取る
