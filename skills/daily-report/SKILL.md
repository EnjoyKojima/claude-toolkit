# デイリーレポート

マージ済みPR・オープンPR・メンバー別着手推奨の3セクションを一気通貫で表示する。

## ワークフロー

### 1. ユーザーへの確認

`AskUserQuestion` ツールで対象期間を聞く:

- 直近1日 (Recommended)
- 直近3日
- 直近7日
- 直近30日

### 2. データ取得

以下の4つのデータを **並列で** 取得する:

#### 2-1. マージ済みPR（指定期間）

```bash
gh pr list --state merged --search "merged:>=YYYY-MM-DD" --json number,title,author,headRefName,baseRefName,mergedAt,body --limit 50
```

#### 2-2. オープン中のPR

```bash
gh pr list --state open --json number,title,author,headRefName,baseRefName,createdAt,isDraft,reviewDecision --limit 50
```

#### 2-3. オープン中のイシュー

```bash
gh issue list --state open --json number,title,assignees,labels,createdAt --limit 100
```

#### 2-4. 直近7日間のクローズ済みイシュー（ペース分析用）

```bash
gh issue list --state closed --json number,title,assignees,closedAt --search "closed:>=YYYY-MM-DD" --limit 100
```

日付は現在日から7日前を計算して指定する。

**注意**: オープンPRのbodyは取得サイズが大きくなるため、一覧では取得しない。サマリーが必要な場合は `gh pr view <number> --json body --jq '.body' | head -30` で個別に取得する。

### 3. 表示

3つのセクションを順に表示する。

---

## セクション1: マージ済みPR

### ヘッダー

```
## 1. 直近X日間にマージされたPR（YYYY-MM-DD〜DD）全N件
```

### 全体サマリー

担当者ごとに1〜2文で **何をやったか** を要約する。PRのbodyから要点を読み取り、個別PRの羅列ではなく活動の全体像が伝わるようにまとめる。

```markdown
### 全体サマリー

- **著者名**: やったことの要約。
- **著者名**: やったことの要約。
```

### 担当者別テーブル

```markdown
### GitHubログイン名 (表示名) — N件

| # | ブランチ → ベース | 内容 |
|---|---|---|
| #123 | `feat/xxx` → `dev` | 変更内容の簡潔な説明 |
```

**テーブルの書き方ルール:**
- `#` 列: `#番号` のみ。リンク不要
- `ブランチ → ベース`: バッククォートで囲む
- `内容`: PRタイトルとbodyから簡潔に1行でまとめる。PRタイトルのprefix（feat:等）は除去

**並び順**: PR件数が多い順。同数の場合はアルファベット順。

---

## セクション2: オープン中のPR

### ヘッダー

```
## 2. オープン中のPR — 全N件
```

### 全体サマリー

担当者ごとに1〜2文で **何に取り組んでいるか** を要約する。

### 担当者別テーブル

```markdown
### GitHubログイン名 (表示名) — N件

| # | ブランチ → ベース | 状態 | 内容 |
|---|---|---|---|
| #123 | `feat/xxx` → `dev` | Open | 変更内容の簡潔な説明 |
| #456 | `fix/yyy` → `dev` | **Draft** | 変更内容の簡潔な説明 |
```

**状態の表記:**
- 通常: `Open`
- ドラフト: `**Draft**`（太字）

**注意事項:**
- PRがスタック（チェーン）されている場合は、テーブル下部に補足で依存関係を記載する
- Bot（copilot-swe-agent等）のPRは別セクションにまとめる

---

## セクション3: メンバー別 今日の着手推奨

### ヘッダーと前提データ

```markdown
## 3. メンバー別 今日の着手推奨

### 前提データ

| 指標 | 値 |
|---|---|
| オープンイシュー総数 | **N件**（うち未アサインM件） |
| 直近7日間のマージPR | **N件** |
| 直近7日間のクローズイシュー | **N件** |
```

### 直近7日間のペーステーブル

マージ済みPR（直近7日分。セクション1の期間が7日未満の場合は別途7日分を取得）とクローズ済みイシューから算出する。

```markdown
### 直近7日間のペース

| メンバー | マージPR | クローズIssue | 1日あたり |
|---|---|---|---|
| ログイン名 | N | ~M | 約X PR/日 |
```

### メンバー別の推奨

メンバーごとにセクションを設け、以下の構成で記載する:

```markdown
### ログイン名 (表示名) — タイプ名

**直近の文脈**: 直近のPR活動から読み取れる作業内容を1〜2文で。

| 優先度 | # | やること | 理由 |
|---|---|---|---|
| 1 | #XXX | 具体的に何を実装・修正するかを平易に説明（例:「画像アップロード時にファイルサイズとMIMEタイプのバリデーションを追加する」） | なぜ今日これをやるべきかの理由 |
| 2 | #YYY | 具体的な作業内容 | 理由 |
| 3 | #ZZZ | 具体的な作業内容 | 理由 |

> 今日の見込みや補足コメント。
```

**「やること」列の書き方ルール:**
- イシュータイトルをそのまま転記しない。「何をどうするか」が伝わる具体的な説明にする
- PRが既にある場合は「PR #NNN をレビュー対応してマージする（〇〇の実装）」のように、アクションを明示する
- 技術用語だけでなく、業務的な意味も添える（例: ×「s3Key重複防止」→ ○「同じ画像ファイルが二重登録されないようにimageCreateにユニーク制約を追加する」）

**タイプ名の付け方**: そのメンバーの直近の作業傾向を端的に表す（例: 「高速消化型」「画像管理集中型」「決済フロー集中型」「管理画面フロントエンド型」など）

**推奨イシューの選定基準**（優先度順）:

1. **オープンPRのマージ推進**: 既にPRが出ているならそのマージ対応が最優先
2. **依存チェーンの解消**: 他メンバーのブロッカーになっているイシュー
3. **文脈の連続性**: 直近のPRと同じ領域のイシュー（コンテキストスイッチが少ない）
4. **アサイン済みイシュー**: 本人にアサインされているもの
5. **スコープの小ささ**: 1日で完了できるサイズのもの

**注意**: 未アサインのイシューでも、メンバーの直近の作業と強く関連するものは推奨に含めてよい。

### チーム全体の所感

最後に、チーム全体を俯瞰した所感を3〜5項目の箇条書きで記載する:

- ボトルネック（レビュー待ち、依存チェーン等）
- クリティカルパス
- 並行着手の機会（パターンが確立された作業の横展開等）

---

## 0件の場合

- マージ済みPRが0件: 「該当期間にマージされたPRはありませんでした」
- オープンPRが0件: 「現在オープン中のPRはありません」
- オープンイシューが0件: セクション3を省略

## MD出力

ユーザーから「MDファイルに出力して」「ファイルに書き出して」等の指示があった場合のみ、全セクションを1つのmdファイルにまとめて出力する。指示がなければ画面表示のみ。
