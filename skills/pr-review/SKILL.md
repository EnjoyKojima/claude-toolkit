# プルリクエスト レビュー

複数の専門エージェントを並列起動してPRを多角的にレビューする。
PRのスコープを厳密に守り、ブロッキングな問題を検出する。
**レビュー後はユーザーと一緒にブロッキング指摘を「この場で修正するもの」と「見送るもの」に振り分け、決まった修正はその場でPRブランチに反映する。**
**Issue化やGitHubへのレビュー投稿（`gh pr review`）は行わない** — 修正と報告のみで完結する。

## 基本思想

**「1つだけ指摘するなら何か」**

レビューの目的は問題の網羅的な列挙ではない。マージをブロックすべき根本的な問題を見つけ出すことである。
改善提案の洪水はPRのスコープを崩壊させる。

**「レビューと修正を分断しない」**

レビューだけして指摘を残すのではなく、ユーザーと一緒にその場でトリアージし、影響範囲の小さい修正は即座に反映する。これにより：
- レビュー → 別セッションで修正、というコンテキスト切替コストを削減
- 「指摘されたが結局やらない」放置を防ぐ
- 修正の妥当性をレビュー文脈そのままで判断できる

### ブロッキング vs アウトオブスコープの判断基準

**ブロッキング（PR内で対応すべき）:**
- このPRが導入するバグ（既存機能の破壊、新機能の不具合）
- このPRの変更によるセキュリティ脆弱性
- このPRで変更されたコードのロジックエラー
- このPRの実装がPR説明・リンク先Issueの要件を満たしていない

**アウトオブスコープ（参考情報として一覧表示するのみ。Issue化やアクションはしない）:**
- このPRが触れていない既存コードの問題
- 「ついでにこれも直したほうがいい」系の改善
- リファクタリング提案
- テストカバレッジの拡充（PRの変更範囲外）
- ドキュメントの不整合（PRの変更範囲外）
- パフォーマンス改善の余地（PRが性能問題を導入していない場合）

## ワークフロー

### 1. PR情報の取得

ユーザーがPR番号を指定しない場合、現在のブランチから推定する。

`gh` コマンドで以下を取得する：
- PR のタイトル、説明文、リンクされたIssue
- PR の差分（全ファイル）— `gh pr diff <number> --repo <owner/repo> > /tmp/pr<number>.diff`
- PR のコミット履歴
- 既存のレビューコメント（あれば）

### 2. スコープの特定

PR説明文とリンク先Issueから、このPRが **何をやろうとしているか** を明確にする。
これがレビューの判断軸になる。スコープは後続のエージェントに共有する。

以下の形式でスコープを整理する：
```
【PRのスコープ】
- 目的: （PR説明文・Issue から読み取った目的）
- 変更範囲: （変更されたファイル・モジュールの一覧）
- やらないこと: （PR説明文に明記されたスコープ外事項）
```

### 3. レビューチームの並列起動

5つの専門エージェントを **Agent ツールで並列起動** する（必ず単一メッセージで複数Agentを並行起動）。

各エージェントには以下の情報を渡す：
- PRのスコープ（ステップ2で整理したもの）
- PR差分ファイルパス（`/tmp/pr<number>.diff`）
- リポジトリのCLAUDE.md パス（あれば）
- レビュー観点固有の指示

各エージェントへの **共通指示** として以下を含める：

```
【レビューの大原則】
あなたはこのPRの差分をレビューする専門レビュワーである。

1. PRのスコープを厳密に守ること。PRが変更していないコードへの指摘はアウトオブスコープとして分類する
2. 発見を以下の2カテゴリに分類すること：
   - 🔴 ブロッキング: このPRの変更が導入する問題（バグ、セキュリティ、要件未充足）
   - 📋 アウトオブスコープ: PRが触れていない既存コードの問題、改善提案、リファクタリング
3. ブロッキング指摘は「1つだけ選ぶなら」の基準で最も重大なものに絞ること
4. 各指摘には具体的なファイルパスと行番号を含めること
5. レビュー結果は日本語で出力すること

【出力形式】
## [観点名] レビュー結果

### 🔴 ブロッキング
（なければ「なし」と記載）
- **問題**: 具体的な問題の説明
- **ファイル**: `path/to/file.ts:行番号`
- **理由**: なぜこれがブロッキングなのか
- **修正案**: 具体的な修正方法

### 📋 アウトオブスコープの発見
（なければ「なし」と記載）
- **発見**: 内容の説明
- **ファイル**: `path/to/file.ts:行番号`
- **推奨Issueタイトル**: Issueとして切り出す場合のタイトル案
- **関連Issue**: 既存のIssueがあればその番号
```

#### チーム1: ロジック正確性レビュワー

```
あなたはロジックの正確性を専門とするレビュワーです。

レビュー観点：
- PRの変更によって導入されるバグはないか
- エッジケースの考慮漏れはないか（null/undefined、空配列、境界値）
- 条件分岐のロジックは正しいか（off-by-one、論理演算子の誤用）
- 非同期処理のエラーハンドリングは適切か
- データ変換の正確性（型変換、日付処理、数値計算）
- PR説明文の要件をコードが正しく満たしているか

やらないこと：
- コードスタイルへの指摘（リンター・フォーマッターの仕事）
- PRが触れていないコードへの指摘
- 「こう書いたほうがきれい」系の提案
```

#### チーム2: セキュリティレビュワー

```
あなたはセキュリティを専門とするレビュワーです。

レビュー観点：
- PRの変更がセキュリティ脆弱性を導入していないか
- 入力バリデーションの漏れ（SQLインジェクション、XSS、コマンドインジェクション）
- 認証・認可チェックの欠落や迂回
- 機密情報の漏洩（ログへの個人情報出力、エラーメッセージでの内部情報露出）
- 安全でない暗号化・ハッシュの使用
- OWASP Top 10 に該当する問題

やらないこと：
- PRが触れていないコードのセキュリティ問題
- 理論的にのみ可能な攻撃シナリオ（実際の脅威モデルを考慮する）
```

#### チーム3: 設計・アーキテクチャレビュワー

```
あなたはソフトウェアアーキテクチャを専門とするレビュワーです。

レビュー観点：
- PRの設計がプロジェクトの既存アーキテクチャと整合しているか
- 責務の分離は適切か（リゾルバーにビジネスロジック、サービス層にDB直接アクセスなど）
- 依存関係の方向は正しいか
- 命名がドメインの概念を正しく反映しているか
- このPRの変更が将来の拡張を不必要に困難にしていないか

CLAUDE.mdがある場合はプロジェクトの規約に照らして判断すること。

やらないこと：
- 全体的なアーキテクチャの改善提案（PRスコープ外）
- 既存コードのリファクタリング提案（PRが触れていない部分）
```

#### チーム4: テスト品質レビュワー

```
あなたはテスト品質を専門とするレビュワーです。

レビュー観点：
- PRの変更に対するテストが存在するか
- テストがPRの変更の重要なケースをカバーしているか
- テストの assertion は正しいか（テストが常に通る偽陽性になっていないか）
- テストが実装の詳細ではなく振る舞いをテストしているか
- テストデータは適切か（エッジケースを含んでいるか）
- 既存テストがPRの変更によって壊れていないか

やらないこと：
- PRが変更していないコードのテスト不足への指摘
- テストカバレッジの網羅的な拡充提案
- テストのコードスタイルへの指摘
```

#### チーム5: データ整合性・副作用レビュワー

```
あなたはデータベースとシステム間連携を専門とするレビュワーです。

レビュー観点：
- PRの変更がデータ整合性を破壊しないか（トランザクション境界、外部キー制約）
- マイグレーションの安全性（データ消失、ロック時間、ロールバック可能性）
- N+1クエリやパフォーマンス劣化の導入
- 外部サービス呼び出しのエラーハンドリング（決済API、認証サービス等）
- 副作用の管理（メール送信、イベント発行、外部API呼び出しのタイミング）
- 冪等性が必要な操作で冪等性が担保されているか

やらないこと：
- 既存のN+1問題やパフォーマンス課題への指摘（PRが悪化させていない場合）
- データモデル全体の改善提案
```

### 4. 結果の統合

全エージェントの結果を収集し、以下のように統合する：

#### 4a. ブロッキング指摘の統合
全チームの 🔴 ブロッキング指摘を集め、重複を排除する。
同じ問題を複数チームが指摘している場合、最も詳細な説明を採用する。

ブロッキング指摘が多すぎる場合（3つ以上）、本当にマージをブロックすべきものだけに絞る。
「1つだけ指摘するなら」のフィルターを適用する。

#### 4b. アウトオブスコープ指摘の整理（参考情報のみ）
アウトオブスコープの発見は **Issue化もアクションもしない**。参考情報としてユーザーに一覧提示するだけ。
- 重要度（致命的 / 改善 / Nice to have）で並べ替える
- 重複を排除し、本質を一行に圧縮する（提示形式はステップ5a 参照）

### 5. ユーザーとのトリアージ（重要）

統合結果をユーザーに提示する。**提示は「本質を一行で」**。

#### 5a. 提示の原則（厳守）

ユーザーが脳内で処理できる短さに圧縮することが目的。次の3つを**ユーザー向けテキストに書かない**：

- **ファイルパス・行番号**（長くて読みづらく、本質の把握を邪魔する）
- **詳細な問題説明**（agent 出力の長文をそのまま貼らない）
- **修正案・修正コード**（トリアージは「やるか/やらないか」を決める場であり、コードを読む場ではない）

各指摘は次の形式の**1行**にまとめる：

```
B1. [本質を表す15〜40字程度の短いタイトル] — [必要なら一文の補足、なくてもよい]
```

例：
- ✅ `B1. 住所スキップフラグが古いクロージャ参照で巻き戻る`
- ✅ `O1. PR説明文の表に書いた未実装機能の扱い`
- ❌ `B1. SimpleOrderClientComponent.tsx の addResolvedGiftToCart 末尾の setSkipAddressModal(... || skipAddressModal) で stale closure が...`（長い・ファイル名混入・詳細）

ファイルパス・行番号・修正コードは**修正実行ステップ（ステップ6）で使うため内部で保持**するが、トリアージ提示には出さない。「詳細が知りたい」と言われた場合のみ追加で展開する。

#### 5b. 質問の構成

`AskUserQuestion` で **ブロッキング指摘の対応方針のみ** を確認する。アウトオブスコープのIssue化やPRレビュー投稿は行わないため、それらの質問はしない。

**Q. ブロッキング指摘の対応方針**
- 選択肢: 「この場で修正する（推奨）」「見送る」の2択
- ブロッキングが0件なら質問をスキップ（修正なし。報告のみで完結）
- 複数件ある場合は本文に B1/B2/... を列挙し、`multiSelect: true` で「どれを修正するか」を聞く（選択しなかったものは見送り扱い）

アウトオブスコープの発見は参考情報として本文に一覧表示するだけで、質問の対象にはしない。

#### 5c. 提示本文のテンプレート

```
## レビュー総評
[1〜2文。何が一番気になるか / 全体の印象]

## 🔴 ブロッキング（N件）
- B1. [本質一行]
- B2. [本質一行]

## 📋 アウトオブスコープ（M件）
- O1. [本質一行]
- O2. [本質一行]
- O3. [本質一行]
...

> 詳細（ファイル箇所・具体的な修正方法）が必要なものがあれば番号で教えてください。
```

テーブル形式（観点列・ファイル列）は使わない。箇条書きの一行リストで十分。

### 6. その場での修正実行

ユーザーが「この場で修正する」と選んだ指摘について、以下の手順で実行する：

#### 6-1. PRブランチへの切替
- 自分が PR 著者か他人かを確認（`gh pr view <n> --json author,headRefName`）
- PRブランチに checkout (`git checkout <headRefName>` → `git pull --ff-only`)
- working tree が clean か確認（dirty なら一旦保留してユーザー確認）

#### 6-2. 修正の実行
- 各指摘について Edit ツールで修正
- スキーマ変更があれば `pnpm codegen`、DBスキーマ変更があれば `pnpm db:generate` / `pnpm db:migrate`
- マイグレーションファイルは直接編集禁止（CLAUDE.md規約準拠）

#### 6-3. 検証
- `pnpm tc`（型チェック）
- `pnpm lint`（リント）
- `pnpm fmt`（フォーマット — コミット前に必ず実行）
- 関連テスト（最低限）：`pnpm --filter <package> test:run -- <test-file>` で対象ファイルのみ
- 全体テスト：必要に応じて `pnpm test:run`
- フロントエンドの stale `.next/dev` キャッシュが TypeScript エラーを起こす場合あり → `rm -rf frontend/<app>/.next` で解消

#### 6-4. コミット & プッシュ
- Conventional Commits 形式で日本語コミットメッセージ
- メッセージはレビュー指摘との対応が分かるように `B1: ...` `B2: ...` のように記載
- Co-Authored-By 行を含める
- `git push origin <branch>`

### 7. 結果の報告

GitHubへのIssue作成・レビュー投稿は行わない。ユーザーへのテキスト報告のみで完結する。

ユーザーに以下を報告する：
- 修正してプッシュした内容（コミットハッシュ、対応した B1/B2/... の対応関係）
- 見送ったブロッキング指摘（後から振り返れるよう本質一行で残す）
- 参考：アウトオブスコープの発見一覧（一行ずつ。アクションは取らない）

## 注意事項

- 各エージェントにはPR差分の全文（あるいはファイルパス）を渡す。省略しない
- エージェントは `subagent_type` を指定せず一般的なエージェントとして起動する（リポジトリのコード読み込みが必要なため）
- 5つのエージェントは全て **並列で** 起動する（単一メッセージに複数 Agent 呼び出し）
- CLAUDE.mdの規約がある場合、それに照らしたレビューを行う
- コードスタイル・フォーマットの指摘は一切しない（リンター・フォーマッターの仕事）
- レビュー結果は日本語で出力する
- **修正実行の前に必ずユーザーの確認を取る** — レビュー → 修正は連続するが、勝手に修正しない
- **Issue化・GitHubへのレビュー投稿（`gh pr review`）はしない** — このスキルはローカルでの修正と報告のみで完結する。アウトオブスコープの発見は参考として一覧表示するだけ
- `AskUserQuestion` の選択肢は1問あたり最大4個。多い場合は質問を分割する
- 修正は基本的に1コミットにまとめる（`fix: PR #N レビュー指摘の修正` 等）。性質が異なる修正は別コミットでもよい
- **トリアージ提示は本質を1行で**。ファイルパス・問題詳細・修正案を最初の提示に含めない（ステップ5a 参照）。詳細はユーザーが番号で指定したときに展開する
- agent には従来通り詳細（ファイル・行番号・問題・修正案）を出させる。アシスタント側で「本質一行」に圧縮してユーザーに提示する
